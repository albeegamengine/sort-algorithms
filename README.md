# Sort Algorithms

バブルソートとクイックソートを実装し、動作確認したプロジェクトです。

# エンジニア課題　概要

Wikipediaの「ソート」に記載されたソートから2個実装してください。
それらを実装したうえで、それぞれそのソートの良かった点を述べてください。
使用する言語や入出力の形式は自由です。
AIは使用せず取り組んでください。
https://ja.wikipedia.org/wiki/%E3%82%BD%E3%83%BC%E3%83%88

提出方法；ファイルの添付もしくは、GitHubリポジトリを共有してください。
期限：一次面接の前日までになります。期限内の実施が難しい場合にはご相談ください。

# 良かった点

## 共通
・バブルソートとクイックソートを選択した理由は基本情報技術者試験に頻出するほど有名なアルゴリズムであるため、この機に実際に実装してみようと思い選択しました。その中でもバブルソートとクイックソートはそれぞれ実行時間が低速/高速と言われており、実際に動作確認を試みました。
・基本設計はバブルソートとクイックソート共通として、配列をコピーしてからソートするようにしました。(参照渡しを防止して元の配列を操作させないようにするため)
・フィールド変数名を統一させました。
・エッジケースとして配列長が1以下の場合を考慮しました。
・動作確認した結果、以下となりました。

| テストケース | バブルソートの実行時間(ms) | クイックソートの実行時間(ms) |
| ---- | ---- | ---- |
| ランダムな配列 | 3 | 1 |
| 既にソートされている配列 | 1 | 1 |
| 逆順の配列 | 1 | 1 |
| 重複する値を含む配列 | 1 | 1 |
| 大きな要素(1000要素)の配列 | 27 | 10 |

## バブルソート
・Wikipediaに記載されている通り、単純なロジックでした。
・ロジックは単純でしたが二重ループしている分、配列の要素が大きくなればなるほどクイックソートよりも時間がかかりました。
・配列の要素数が10の場合、最良計算時間(ソート済の配列)および最悪計算時間(逆順の配列)の比較にならなかった一方で、ランダムな配列が最も実行時間が長いという結果になりました。

## クイックソート
・ロジックはバブルソートよりも複雑ですが、Wikipediaに他のソート法と比べて最も高速だと言われている通り、少なくともバブルソートよりも実行時間が短く、バブルソートの方が約2.7倍の実行時間となりました。
・配列の要素数が10の場合、最良計算時間(ピボットが中央値に近い場合)および最悪計算時間に近いケース(ピボットが最大値または最小値を取り続ける場合)はいずれも1msと比較になりませんでした。
